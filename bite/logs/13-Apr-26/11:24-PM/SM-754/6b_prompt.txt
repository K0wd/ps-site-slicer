# Automation Conventions

## General QA Behavior

- Always double check your work.
- Apply the KISS principle: choose the simplest implementation that is clear, readable, and sufficient for the scenario.
- Prefer deterministic waits such as `waitForURL`, `waitForLoadState`, and visible-ready checks instead of sleeps.
- Keep Playwright execution cross-platform by using npm scripts only and avoiding OS-specific shell syntax.

## Project Structure

- Use TypeScript for automation support code, hooks, step files, and reusable properties.
- Keep this structure aligned with the repo:

```text
features/
|-- *.feature
|-- steps/
|   `-- <pageName>.steps.ts
`-- support/
properties/
`-- *.ts
```
- Put shared world and hooks in `features/support/`.
- Put Gherkin scenarios in `features/*.feature`.
- Put all step logic in `features/steps/<pageName>.steps.ts`.

## Gherkin And Scenario Writing

- Keep Gherkin scenarios readable and business-focused; keep browser mechanics inside reusable step files and support files.

## Data And Reusable Properties

- Put test data or reusable variables in `properties/`.
- Store every static value as a variable in that page's `properties/<pageName>.properties.ts` file.
- Global time format should be: YYYY-MM-DD HH:MM AM/PM

## Selectors And Locator Design

- Define page locators as XPath variables in that page's `properties/<pageName>.properties.ts` file.
- Prefer plain exported string constants and `Record<string, string>` maps for locator lookups; avoid `keyof typeof` casting for runtime element-name mapping.
- Use XPath-only selectors for element targeting in automation steps.
- For simple locator lookups, prefer direct map access with descriptive local variable names instead of trivial resolver helpers.

## Self-Healing Mechanism

- Add self-healing only for locator drift (same behavior, changed DOM shape); do not self-heal business logic failures, permission failures, or assertion mismatches.
- Model self-heal selectors as ordered XPath fallback arrays in `properties/<pageName>.properties.ts` (primary first, then stable fallback options).
- In step files, attempt selectors in order and stop on first unique match (`count === 1`); if none match uniquely, fail fast with a clear error.
- Log every fallback usage (primary failed, fallback used) and gate verbose self-heal logs behind an explicit `.env` flag.
- Keep fallback chains short (max 2-3 alternates) and remove stale selectors during maintenance to avoid masking real UI regressions.
- For any new self-healed selector path, add or update a regression scenario that validates both normal-path behavior and one negative-path assertion.

## HTML Snapshot Investigation Rule

- Always check `html/` first for an existing page `htmlBody` snapshot before capturing a new one.
- If the page snapshot does not exist in `html/`, capture `htmlBody` and save it in `html/` using a deterministic page-based filename with no timestamp suffix.
- Capture snapshots especially when adding/updating XPath selectors or when export/download controls are missing, delayed, or detached.
- Use saved `htmlBody` artifacts to refine XPath selectors, then keep only stable selectors in properties files.
- Do not keep snapshot capture as noisy default logging for every unrelated step; use it where investigation value is high.

## XPath Learning Rule

- When adding or updating checks on a page, first inspect the page `htmlBody` snapshot in `html/`; if missing, capture it and save to `html/` before finalizing XPath locators.
- Build XPath selectors from stable anchors in this order: `data-automation-id` -> semantic attributes (`title`, `role`, `aria-*`) -> stable heading/label text.
- For any newly learned locator, keep a primary XPath plus up to 2 fallback XPaths when DOM shape is known to drift.
- Validate learned locators with at least one positive assertion scenario on the same page update.
- If a learned XPath cannot be made stable, fail fast with a clear message and keep investigation snapshots for follow-up instead of masking the issue.
- For sidebar/menu XPath updates, use the captured `htmlBody` to confirm the exact clickable node and the exact expanded-state container before adding retries.
- When a sidebar item exists in a scrollable container, scroll the sidebar container or `scrollIntoViewIfNeeded()` for the exact locator before introducing alternate XPath guesses.
- Prefer one verified semantic XPath (for example `@title` + stable `@href`) over broad union XPath retries; only keep fallback XPath variants when the HTML snapshot shows real DOM drift across states.

## Assertions And Validation

- Ensure each XPath used for assertions resolves to exactly one element (`count === 1`) before interacting or validating.
- For table-driven element checks, map human-readable element names to XPath variables in the page properties file.
- Keep assertion error messages short and reusable; prefer shared constants over repeating long literal strings.
- Use Chai assertions that validate behavior clearly and avoid brittle page text checks when a stable page-level signal exists.

## Reusable Export And Data Flows

- For dashboard-vs-detail export validations, reuse one KISS flow: `Navigate -> Search -> Export -> View Exports -> Refresh until ready -> Download CSV -> Parse -> Capture metric -> Return to Dashboard -> Compare`.
- Keep export/download mechanics reusable and place only metric-specific logic in dedicated steps.

## Logging And Reporting

- Gate verbose scenario logs with explicit `.env` flags; avoid always-on noisy console output.
- For CSV-driven debug logs, print a stable and grep-friendly format in this format:
  - `Scenario Name: <scenario name>`
  - `| inventory id | related column data (this can be multiple) |`
- Keep debug table values sanitized for plain-text logs (replace `|` in cell text) and source them from stable CSV headers.
- Keep command-line reporting simple and readable when updating Cucumber configuration.

## Pipelines related rules

- For Azure DevOps feature execution, use one stage per feature file so the pipeline view stays clear and presentation-friendly.
- Do not collapse multiple feature files into a single grouped stage when the intent is feature-level visibility.
- For single-feature stage execution, pass `FEATURE_FILE` as an environment variable and run `npm run test` (avoid passing feature paths as CLI positional arguments).

## Claude CLI Usage For Automation Scripting

- Use `claude -p "<prompt>"` (print mode) for non-interactive, scriptable one-shot prompts that print the response and exit.
- Pipe file content as context: `cat ticket.txt | claude -p "create a test plan for this ticket"`.
- Pipe command output as context: `echo "<details>" | claude -p "<prompt>"`.
- Use `--output-format json` for machine-parseable output in scripts: `claude -p --output-format json "<prompt>"`.
- Use interactive mode (no `-p`) for conversational follow-ups: `claude "your question"`.
- In shell scripts, pass variables directly:
  ```bash
  TICKET="ticket description here"
  claude -p "Create a test plan for: $TICKET"
  ```
- Use `claude -p --allowedTools Read,Grep,Glob "<prompt>"` to limit which tools Claude can use in non-interactive mode.
- Use `claude -p --model <model-id> "<prompt>"` to target a specific model (e.g. `claude-sonnet-4-6` for faster/cheaper scripted tasks).
- Use `claude -c` to continue the most recent conversation.
- Use `claude -r <session-id>` to resume a specific past session.

## Scope And Risk

- Keep shared hooks/world in `features/support/`.
- Keep reusable constants and XPath locators in `properties/*.ts`.
- Use TypeScript for support, hooks, and step files.

## Navigation And Waiting

- Use resilient sidebar locators anchored by stable attributes such as `@title`.
- Ensure parent menus are expanded before clicking submenu links.
- Prefer normal click, fallback to `dispatchEvent("click")` only when needed.
- Use `scrollIntoViewIfNeeded()` before click attempts on sidebar items.
- Assert `waitForURL(...)` after navigation clicks.
- Prefer `waitForURL`, `waitForLoadState`, and visible/attached checks over sleeps.
- In headless runs, keep a desktop viewport for dashboard stability.

- Use XPath selectors in page property files.
- Use plain constants or `Record<string, string>` maps for locator lookup.
- Self-heal only locator drift, not business-logic/assertion failures.
- Model self-heal with ordered fallback XPaths (2-3 alternates max).
- Resolve selector candidates by first unique match (`count === 1`), otherwise fail fast.
- Log fallback usage behind explicit `.env` flags.
- Remove stale fallback selectors during maintenance.

- Ensure assertion target XPaths resolve to expected uniqueness before interaction.
- Keep assertion messages short, clear, and reusable.
- For table-driven checks, map user-facing names to property-file locator keys.
- Add at least one failure-path assertion when adding new positive flow coverage.

## Export Flow Standards

- Reuse a consistent flow for export validation:
- `Page Ready -> Export -> View Exports -> Refresh until ready -> Download -> Parse -> Assert data`.

- Update `changelogs.md` for coverage/behavior/process changes.
- Keep changelog newest-first.
